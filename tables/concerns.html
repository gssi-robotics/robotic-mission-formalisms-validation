Summary of model expressivity for mission concerns (● full support; ◐ partial support; ○ no support)
Mission concern BT SM HTN BPMN Rationale
Reactive behavior ● ● ○ ● BT and SM leverage their reactive nature and event-handling structures. BPMN features boundary events and event sub-processes. HTN does not support it.
Decision making ● ● ● ● SM and BPMN feature conditional control structures for runtime decision-making. In BTs this is realized by combining fallbacks and sequence nodes. HTN relies on planning according to pre- and post-conditions.
Time-dependent behavior ◐ ◐ ○ ● In BT and SM it must be realized by manually implementing the control of timing through condition nodes and state implementations. BPMN features timer events attached to tasks. HTN does not support it.
Task status ● ● ○ ● BT, SM, and BPMN support it through the tick return value, the events outgoing from a state, and the task activity lifecycle, respectively. HTN does not support it.
Robot-robot interaction ◐ ◐ ◐ ● BT, SM, HTN rely on the task implementation. BPMN has dedicated structures, i.e., message and signal events for communication.
Human-robot interaction ◐ ◐ ◐ ● BT, SM, HTN rely on the task implementation. BPMN has a dedicated task type (user task).
Robot-external system interaction ◐ ◐ ◐ ● BT, SM, HTN rely on task implementation. BPMN features service tasks, send/receive tasks, and message events.
State saving and task resuming ◐ ◐ ○ ○ BT relies on conditions checked at each tick and shared knowledge (blackboards). SM requires ad hoc states and events to pause and resume. HTN and BPMN do not support it.
Explicit waiting ◐ ◐ ◐ ● BT, SM, and HTN rely on ad hoc implementation of actions, states, or tasks. BPMN supports timer events.

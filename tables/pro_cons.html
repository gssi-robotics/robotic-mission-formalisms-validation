Overview of formalism expressivity strengths and weaknesses
Strengths Weaknesses
BT
  • Easily express reactive behavior when the robot has to continuously react to changing conditions.
  • Clear task status semantics, making progress and failure handling a first-class concern.
  • The semantics Failure is overloaded, requiring additional disambiguation through ad hoc nodes.
  • Difficult to trace the origin of failures returned by deeper nodes without explicit checking.
  • No standard way for expressing waiting, temporal constraints, and interactions, which must be manually implemented.
  • The use of nodes with memory for tracking mission execution state hinders reactivity by limiting reevaluation.
SM
  • Natively models event-driven behavior defining how the system should react to events and transition between states.
  • Easily handles task status as distinct states and via transitions outgoing from them.
  • Handling errors and events requires outgoing transitions from all potentially affected states, requiring explicit transitions for system-level events.
  • Task resuming requires ad hoc selector states and transitions.
  • Interactions, messaging, and coordination with robots, systems, or humans must be manually realized.
  • Time-dependent behaviors require manual integration of timers.
HTN
  • Provides support for decision making through runtime planning by decomposing tasks using methods.
  • Natively allows explicit definition of pre- and post-conditions.
  • No expressiveness for reactive behavior, which must be handled outside the mission model.
  • No explicit expression of static choices, always delegated to planning.
  • No task-status tracking, delegated entirely to the HTN executor.
  • Interactions, waiting, and time-driven behaviors are not supported and must be manually implemented.
BPMN
  • Decision-making logic is easy to model thanks to the process-oriented structure.
  • Rich notation explicitly supporting diverse event and error handling making it very expressive for time-dependent behavior, event/error handling, and waiting.
  • Explicitly allows modeling of humans activities and external systems, and offers a clear representation of robot-robot, human-robot and robot-system interactions.
  • No native support for specifying task or mission resuming after interruption.
  • Not optimal for reactions to highly frequent events.
  • Can become complex for detailed or extended robotic missions.

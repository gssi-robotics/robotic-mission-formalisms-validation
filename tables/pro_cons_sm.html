Strengths Weaknesses
  • Natively models event-driven behavior defining how the system should react to events and transition between states.
  • Easily handles task status as distinct states and via transitions outgoing from them.
  • Handling errors and events requires outgoing transitions from all potentially affected states, requiring explicit transitions for system-level events.
  • Task resuming requires ad hoc selector states and transitions.
  • Interactions, messaging, and coordination with robots, systems, or humans must be manually realized.
  • Time-dependent behaviors require manual integration of timers.

SMs employs transitions from states to explicitly distinguish between different events and different outcomes of the actions executed by the robots within each state. This allows the mission model to unambiguously distinguish different events and error causes, and to explicitly redirect the flow towards the states modeling their handling.
As a drawback, handling events that can potentially affect every state require outgoing transitions from each of them, hence making the model more complex. Organizing states hierarchically allows reducing such complexity. Similarly, for resuming a temporarily interrupted mission, transitions towards all the possible interrupted states have to be modeled. This also requires the specification of ad hoc selector states, hence contributing to the model complexity.
The handling of temporal-related behaviors has to be manually realized within the single state logic, as well as the interaction with other robots, systems, and humans.
